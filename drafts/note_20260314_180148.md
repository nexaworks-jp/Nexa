# Claude Codeで仕様駆動開発に挑戦｜テストファーストでコード品質を上げる方法

# Claude Codeで仕様駆動開発に挑戦｜テストファーストでコード品質を上げる方法

## リード

「テストを先に書く」って聞いたことありますか？わたしも最初は「え、本体コードを書く前にテスト？なんだか逆じゃない？」と思っていました。

でも、Claude Codeを使って仕様駆動開発（テストファーストの開発方法）に挑戦してみたら、**「あ、これってコードの品質がぜんぜん違う」**ってびっくりしました。

今日はわたしが実際に試してみてよかった、Claude Codeでテストを先に書く流れをシェアします。小規模なプロジェクトから始めたい方、コードの品質に悩んでいる方に向けた内容です。

---

## なぜ「テストファースト」が必要なのか

### 背景：コードの問題が後から見える

わたしが気づいたのは、**通常の流れだと、バグが見つかるのはずっと後**だということです。

通常の開発フロー：
1. 仕様を読む
2. コードを書く
3. テストを書く（あるいは手動テスト）
4. バグが見つかる
5. コードを修正
6. テストを修正

これだと、修正のコストが何倍にもなります。

### テストファースト開発（TDD）の考え方

テストファーストは逆です：

1. **仕様を理解する**
2. **テストを先に書く**（仕様を「テストコード」として表現する）
3. テストが失敗することを確認する（Red）
4. テストを通すためのコードを書く（Green）
5. コードを整理する（Refactor）

このRed → Green → Refactorのサイクルを「Red-Green-Refactorサイクル」と呼びます。

### Claude Codeを使うメリット

Claude Codeが優秀な点は、**テストコードを読み込んで、それを通すコードを自動で生成できる**ということです。仕様を曖昧に説明するより、テストコードを見せた方が、Claudeは「あ、こういう動作を期待してるんだ」と正確に理解します。

## 実際のファイル構成とコード例

### 1. プロジェクト構成

以下のような構成を想定します：

```
my-tdd-project/
├── src/
│   └── calculator.js        # 本体コード
├── tests/
│   └── calculator.test.js   # テストコード
├── package.json
└── README.md
```

### 2. テストコードを先に書く

まずは、テストファイル（`tests/calculator.test.js`）を作ります。わたしたちが作りたいのは「シンプルな計算機」です。

```javascript
// tests/calculator.test.js
const { add, subtract, multiply, divide } = require('../src/calculator');

describe('Calculator', () => {
  describe('add', () => {
    test('should return 5 when adding 2 and 3', () => {
      expect(add(2, 3)).toBe(5);
    });

    test('should handle negative numbers', () => {
      expect(add(-2, 3)).toBe(1);
    });

    test('should handle zero', () => {
      expect(add(0, 5)).toBe(5);
    });
  });

  describe('subtract', () => {
    test('should return 2 when subtracting 3 from 5', () => {
      expect(subtract(5, 3)).toBe(2);
    });
  });

  describe('multiply', () => {
    test('should return 12 when multiplying 3 and 4', () => {
      expect(multiply(3, 4)).toBe(12);
    });

    test('should return 0 when one factor is zero', () => {
      expect(multiply(5, 0)).toBe(0);
    });
  });

  describe('divide', () => {
    test('should return 3 when dividing 12 by 4', () => {
      expect(divide(12, 4)).toBe(3);
    });

    test('should throw an error when dividing by zero', () => {
      expect(() => divide(10, 0)).toThrow('Cannot divide by zero');
    });
  });
});
```

このテストコードだけで、わたしたちが求めている仕様が明確に見えます。

### 3. Claude Codeに見せる

Claude Codeの「Code」ボタンを開いて、このテストコードを貼り付けて指示します：

**プロンプト例：**

```
このテストコードを読んでください：

[上記のテストコードを貼り付け]

このテストをすべて通すJavaScriptコードを src/calculator.js に書いてください。
テストの要件をすべて満たすコードを生成してください。
```

Claude Codeは、このテストを読み込んで、こんなコードを生成します：

```javascript
// src/calculator.js
function add(a, b) {
  return a + b;
}

function subtract(a, b) {
  return a - b;
}

function multiply(a, b) {
  return a * b;
}

function divide(a, b) {
  if (b === 0) {
    throw new Error('Cannot divide by zero');
  }
  return a / b;
}

module.exports = { add, subtract, multiply, divide };
```

## ステップバイステップの実装手順

### ステップ1：プロジェクトのセットアップ

```bash
# 新しいプロジェクトディレクトリを作成
mkdir my-tdd-project
cd my-tdd-project

# npm を初期化
npm init -y

# Jest（テストランナー）をインストール
npm install --save-dev jest
```

`package.json` に test スクリプトを追加：

```json
{
  "scripts": {
    "test": "jest"
  }
}
```

### ステップ2：テストコードを書く

`tests/calculator.test.js` を作成し、上のテストコード例を貼り付けます。

この段階では、本体コード（`src/calculator.js`）はまだ存在しません。

### ステップ3：テストが失敗することを確認（Red）

```bash
npm test
```

こう出ます：

```
FAIL  tests/calculator.test.js
  ● Cannot find module '../src/calculator'
```

これが「Red」の状態です。**これは正常です**。テストが失敗するのが期待値です。

### ステップ4：Claude Codeでコードを生成（Green）

Claude Codeを開いて、上のプロンプトを実行します。生成されたコードを `src/calculator.js` に保存。

### ステップ5：テストが通ることを確認

```bash
npm test
```

出力：

```
PASS  tests/calculator.test.js
  Calculator
    add
      ✓ should return 5 when adding 2 and 3
      ✓ should handle negative numbers
      ✓ should handle zero
    subtract
      ✓ should return 2 when subtracting 3 from 5
    multiply
      ✓ should return 12 when multiplying 3 and 4
      ✓ should return 0 when one factor is zero
    divide
      ✓ should return 3 when dividing 12 by 4
      ✓ should throw an error when dividing by zero
```

これが「Green」です。すべてのテストが通りました。

### ステップ6：コードをきれいにする（Refactor）

コードが動くことが確認できたら、必要に応じてリファクタリングします。

テストが通っているので、安心して修正できます。

## よくあるつまずきと解決策

### つまずき1：「Claude Codeが部分的なコードしか生成しない」

**原因**：プロンプトが曖昧だったり、テストコードが不完全な場合があります。

**解決策**：

```
「以下のテストコードをすべてパスするコードを生成してください。
出力は完全な動作するコードで、require文を含めてください。

[テストコード全体を貼り付け]
```

と明確に指示しましょう。

### つまずき2：「Jest がインストールされていないとエラーが出る」

**原因**：`npm install` を忘れた。

**解決策**：

```bash
npm install --save-dev jest
```

を実行してください。

### つまずき3：「テストファーストだと時間がかかるんじゃ？」

**これが重要なポイント**です。わたしも最初はそう思いました。

でも、実際には：
- テストが「設計ドキュメント」になるので、後で修正するときが早い
- バグが見つかるのが早いので、全体の開発時間は短くなる傾向
- チーム開発のときに特に効果的（誰が書いたコードでも、テストを見れば動作が明確）

## 応用：複雑な機能をテストファーストで実装する

テストファーストは計算機だけじゃなく、もっと複雑な機能にも使えます。

例えば「ユーザー認証」をテストファーストで設計する場合：

```javascript
// tests/auth.test.js
const { authenticate, hashPassword } = require('../src/auth');

describe('Authentication', () => {
  test('should authenticate with correct password', () => {
    const hash = hashPassword('mypassword123');
    expect(authenticate('mypassword123', hash)).toBe(true);
  });

  test('should reject incorrect password', () => {
    const hash = hashPassword('mypassword123');
    expect(authenticate('wrongpassword', hash)).toBe(false);
  });

  test('should not expose plain password in hash', () => {
    const hash = hashPassword('mypassword123');
    expect(hash).not.toContain('mypassword123');
  });
});
```

このテストコードを Claude Code に見せれば、セキュアなハッシュ関数を実装してくれます。
